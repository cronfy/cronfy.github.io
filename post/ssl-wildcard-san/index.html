Создание wildcard-сертификата SSL для разработки с SAN и установка его в Google Chrome 58+

June 16, 2017

3 minute read

  • Twitter

  • Twitter

  • Google+

  • Facebook

  • LinkedIn

  • Email

Удобно, когда на компьютере разработчика сайты работают по HTTPS. Это позволяет отлаживать все, что требует защищенного соединения в браузере, не прибегая к каким-либо хакам.

Удобно также, когда для каждого нового сайта не нужно создавать новый сертификат, а есть один сертификат для всех сайтов.

Существует много инструкций по созданию сертификатов, но начиная с версии Chrome 58 многие из них не работают. Это связано с тем, что теперь сертификаты с указанием имени домена через Common Name не поддерживаются. Вместо этого нужно использовать SAN (Subject Alternative Name).

Инструкция о том, как это сделать, взята с сайта Alexander Zeitler.

Что потребуется:

  • Создать корневой сертификат центра сертификации (CA).
  • С помощью этого сертификата выпустить wildcard-сертификат с использованием SAN для имени вида *.dev.loc (все поддомены третьего уровня). Домен должен быть обязательно третьего уровня или больше (домен второго уровня работать не будет).
  • Установить корневой CA сертификат в Google Chrome.
  • Для всех локальных сайтов с SSL использовать имя вида *.dev.loc, например, example.dev.loc.
  • Прописать wildcard-сертификат в конфигурации виртуальных хостов веб-сервера.

После этого при входе по HTTPS Chrome будет показывать, что соединение “Надежное”.

Создать корневой сертификат центра сертификации (CA)

Нужно создать скрипт createRootCA.sh следующего содержания:

#!/usr/bin/env bash

mkdir ~/ssl/
openssl genrsa -des3 -out ~/ssl/rootCA.key 2048
openssl req -x509 -new -nodes -key ~/ssl/rootCA.key -sha256 -days 1024 -out ~/ssl/rootCA.pem

И запустить его.

Выпустить wildcard-сертификат с использованием SAN

Теперь нужно создать скрипт для генерации сертификата и два вспомогательных файла:

Скрипт createselfsignedcertificate.sh:

#!/usr/bin/env bash

sudo openssl req -new -sha256 -nodes -newkey rsa:2048 \ 
    -out server.csr \
    -keyout server.key \ 
    -config <( cat server.csr.cnf )

sudo openssl x509 -req -days 500 -sha256 -CAcreateserial \ 
    -CA ~/ssl/rootCA.pem -CAkey ~/ssl/rootCA.key \ 
    -in server.csr \
    -out server.crt \
    -extfile v3.ext

Файл server.csr.cnf:

[req]
default_bits = 2048
prompt = no
default_md = sha256
distinguished_name = dn

[dn]
C = US
ST = New York
L = Rochester
O = End Point
OU = Testing Domain
emailAddress = your-administrative-address@your-awesome-existing-domain.com
CN = *.dev.loc

Файл v3.ext, в котором мы и прописываем Subject Alternative Name (параметр @alt_names).

authorityKeyIdentifier = keyid,issuer
basicConstraints = CA:FALSE
keyUsage = digitalSignature, nonRepudiation, keyEncipherment, dataEncipherment
subjectAltName = @alt_names

[alt_names]
DNS.1 = *.dev.loc

И теперь запускаем createselfsignedcertificate.sh.

Установить сертификат в Google Chrome

После запуска первого скрипта создается файл ~/ssl/rootCA.pem. Его нужно добавить в Chrome следующим образом:

  1. Настройки.
  2. (внизу) Показать дополнительные настройки.
  3. HTTPS/SSL -> Настроить сертификаты.
  4. Центры сертификации -> Импорт.
  5. Выбираем файл ~/ssl/rootCA.pem.
  6. Отмечаем галочку “Доверять этому сертификату при идентификации сайтов” и нажимаем OK.

Результатом будет добавленный сертификат центра сертификации “Internet Widgits Pty Ltd”:

Установить сертификат на веб-сервер

В результате работы второго скрипта в текущей директории появятся файлы server.key и server.crt. Их нужно подключить к веб-серверу. В Apache это происходит примерно так:

<VirtualHost *:443>
        ServerName tests.dev.loc

        SSLEngine on
        SSLCertificateFile /var/www/ssl-wildcard/server.crt
        SSLCertificateKeyFile /var/www/ssl-wildcard/server.key

        ...
</VirtualHost>

Результат

Перезапускаем веб-сервер, и теперь сайты работают по SSL.
